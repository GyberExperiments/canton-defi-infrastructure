## 1. Контекст проекта

### 1.1 Что это за проект

- **Название:** cantonnet-omnichain-sdk  
- **Роль:** (1) Мост между Canton Network (Daml ledger) и BNB Smart Chain (BSC, EVM) для двустороннего перевода активов; (2) Rust SDK для работы с Canton Ledger API v2 и смежными сервисами (wallet, crypto, transport, observability).
- **Стек:** Rust (workspace), PostgreSQL, Daml (Canton), Solidity (BSC), Docker, при необходимости Kubernetes.
- **Ключевые потоки:** Canton→BSC (lock на Canton → unlock/mint на BSC), BSC→Canton (lock/burn на BSC → unlock на Canton). Оркестратор координирует потоки, персистентность и recovery.

## 3. Измерения анализа (Best Practices 2025)

Проверить и оценить по каждому пункту: соответствует / частично / не соответствует / не применимо. Для «частично» и «не соответствует» — явно указать, чего не хватает и что сделать.

### 3.1 Разделение слоёв и границы

- Чёткое разделение: domain (типы, ФСМ, правила) / application (orchestrator, use cases) / infrastructure (DB, RPC, gRPC, контракты).
- Зависимости только внутрь: infrastructure → application → domain (domain без зависимостей от БД, ethers, tonic).
- Публичный API крейтов минимален и стабилен; внутренние детали не протекают наружу.
- Bridge-домен (bridge-core) не зависит от canton-ledger-api и ethers; конвертация на границах (canton-client, bsc-client).

### 3.2 Управление зависимостями (Rust 2025)

- Workspace: единые версии ключевых зависимостей через `[workspace.dependencies]`; отсутствие дублирования версий одних и тех же крейтов.
- MSRV и edition: явный `rust-version` и `edition = "2021"` (или 2024 по возможности); совместимость с текущим rust-toolchain.
- Тяжёлые/опциональные зависимости: за feature flags (например `proto_compiled`, `otel`); сборка без лишних фич по умолчанию.
- Cargo-deny / audit: использование deny.toml (bans, duplicates, unmaintained); учёт известных уязвимостей.

### 3.3 Ошибки и надёжность

- Единая таксономия ошибок (thiserror): доменные коды, обёртки для Canton/BSC/DB без потери контекста.
- Граница «транзиентная vs перманентная» ошибка — явно (для retry и перехода в Failed/Stuck).
- Идемпотентность: command_id (Canton), processedTransfers (BSC), уникальность в БД; отсутствие двойного списания/выдачи.
- Circuit breaker и retry с backoff на всех внешних вызовах (Canton gRPC, BSC RPC); параметры в конфиге.

### 3.4 Асинхронность и потоки данных

- Явная модель потоков: Canton stream, BSC subscription, worker pending, finality worker; без блокирующих вызовов в hot path.
- Graceful shutdown: SIGTERM → останов приёма → ожидание in-flight → сохранение checkpoint → закрытие соединений.
- Отсутствие неограниченных очередей; bounded backpressure при высокой нагрузке.

### 3.5 Безопасность

- Секреты только из env/Vault; отсутствие хардкода ключей и паролей.
- Контракты (Daml, Solidity): явные права (signatory/controller, AccessControl), Pausable, ReentrancyGuard, дедуп (processedTransfers), проверка баланса перед переводом.
- Угрозы (rogue operator, скомпрометированный participant, утечка BSC-ключа, griefing) описаны в security.md и учтены в дизайне/runbook.

### 3.6 Наблюдаемость (Observability 2025)

- Структурированное логирование (JSON), единый trace_id на перевод, связка с transfer_id и status.
- Метрики Prometheus: счётчики/гаужи по переводам, задержки RPC, ошибки; метки (direction, status) без кардинальности взрыва.
- Трейсинг: распространение контекста (tracing), при необходимости экспорт в OTLP.
- Health: раздельная проверка Canton, BSC, DB; 503 при недоступности критичного компонента.

### 3.7 Cross-chain и финальность

- Чёткая модель финальности: Canton (offset, один раз обработан); BSC (required_confirmations, учёт reorg).
- Checkpoint offset для Canton хранится в БД; восстановление после рестарта без повторной обработки одного и того же события.
- Состояния перевода и переходы между ними формализованы (ФСМ); нет «подвисших» статусов без политики recovery.

### 3.8 Персистентность и аудит

- Схема БД: нормализованные таблицы, уникальность по ключам идемпотентности (canton_command_id, bsc_tx_hash где применимо), индексы под запросы pending и по времени.
- Append-only аудит смены статусов (transfer_audit_log); при необходимости — экспорт для compliance.
- Миграции только вперёд (SQLx migrate); версионирование миграций.

### 3.9 API и контракты

- REST: версионирование (/api/v1), пагинация (cursor), единый формат ошибок; OpenAPI 3.0 описан и доступен.
- Защита админ-эндпоинтов (retry/cancel/force-complete): auth по API key или JWT.
- Контракты Daml: ключи контрактов, архивирование при CompleteTransfer; Solidity: интерфейсы и тесты (Hardhat).

### 3.10 Тестирование и качество

- Unit-тесты домена (ФСМ, типы, расчёт комиссий); интеграционные тесты БД и при необходимости Canton Sandbox / BSC fork.
- E2E: минимум один сценарий в каждую сторону (Canton→BSC, BSC→Canton).
- CI: сборка, тесты, линт (clippy), при необходимости audit; стабильность сборки workspace.

### 3.11 Деплой и операции

- Конфиг по окружениям (dev/prod); секреты из env/Vault; параметры recovery, circuit breaker, таймаутов в конфиге.
- Docker: multi-stage сборка, минимальный образ; docker-compose для dev.
- Runbook: backup/restore БД, ротация ключей (BSC, Canton JWT), действия при падении participant/BSC RPC и при переполнении очереди.

### 3.12 Документация и воспроизводимость

- Архитектура и решения зафиксированы в docs и в промптах (READY_PROMPT); новый разработчик или модель может воспроизвести мост по документам.
- README и конфиг DevNet (эндпоинты, примеры) актуальны; зависимости между документами явные.

## 4. Формат выходного отчёта

Выдать структурированный отчёт в следующем виде (можно на русском или английском).

### 4.1 Резюме (Executive Summary)

- 1–2 абзаца: текущее состояние архитектуры, главные сильные стороны и главные риски/разрывы.
- Итоговая оценка: в целом соответствует / частично соответствует / не соответствует best practices 2025 по перечисленным измерениям.

### 4.2 Текущее состояние (факт)

- Структура workspace и крейтов (таблица или список); граф зависимостей крейтов (кто от кого зависит).
- Какие части из READY_PROMPT уже реализованы, какие отсутствуют или отличаются.
- Кратко: конфиг, контракты (Daml/Solidity), скрипты, CI, документация.

### 4.3 Разрывы по измерениям (Gaps)

Для каждого из пунктов 3.1–3.12:
- Оценка: соответствует / частично / не соответствует / N/A.
- Конкретные разрывы (что именно не так или чего не хватает), со ссылками на файлы/разделы документов где применимо.

### 4.4 Приоритизированные рекомендации

- Список рекомендаций с приоритетом: P0 (критично), P1 (важно), P2 (желательно).
- Для каждой: что сделать, в каком артефакте (крейт/файл/документ), краткое обоснование.
- По возможности — порядок внедрения (что делать сначала из-за зависимостей).

### 4.5 План рефакторинга (опционально)

- Если нужны структурные изменения (разделение крейтов, перенос слоёв, смена границ): пошаговый план без breaking change там, где это возможно; явно отметить этапы с breaking changes.
- Оценка объёма: мелкий / средний / крупный по каждому блоку.

### 4.6 Чеклист соответствия (сводная таблица)

Таблица: измерение (3.1–3.12) | оценка | ключевые действия. Это даёт быстрый обзор и основу для трекинга исправлений.

## 5. Ограничения и правила

- Опираться только на **реально прочитанное** из репо и перечисленных документов; не приписывать проекту то, чего нет в коде или в документах.
- Рекомендации должны быть **реализуемыми**: конкретные файлы/крейты/конфиги, а не общие фразы.
- Учитывать **контекст моста и SDK**: institutional-grade Canton, BSC mainnet/testnet, необходимость аудита, recovery и observability.
- Если по какому-то измерению информации в репо недостаточно — явно указать «требуется проверка в коде/документах» и что именно проверить.
- Ссылаться на актуальные best practices 2025 (Rust, cross-chain bridges, observability, security); при расхождении с устаревшими подходами — кратко пояснить.
