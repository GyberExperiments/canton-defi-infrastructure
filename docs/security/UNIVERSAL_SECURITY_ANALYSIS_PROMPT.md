## 📐 МЕТОДОЛОГИЯ: S.R.A.P. (Systematic Root Analysis Protocol)

### **S** - System Map (Карта системы)

**Вопросы для анализа:**
```
1. Что защищаем? (Assets - данные, средства, репутация)
   - Список критичных активов системы
   - Ценность каждого актива для атакующего
   - Последствия компрометации

2. Кто может атаковать? (Threat Actors)
   - Внутренние: разработчики, админы, инсайдеры
   - Внешние: хакеры, конкуренты, организованные группы
   - Системные: автоматизированные боты, скрипты

3. Где границы доверия? (Trust Boundaries)
   - Пользователь → Frontend
   - Frontend → Backend API
   - Backend → Database/External Services
   - Microservices → Microservices
   - Internal → External networks

4. Как данные текут? (Data Flow)
   - Откуда приходят данные (sources)
   - Где обрабатываются (processing)
   - Где хранятся (storage)
   - Где передаются (transmission)
   - Куда уходят (destinations)
```

**Действие:** Создай ментальную/текстовую карту системы с этими 4 слоями.

---

### **R** - Risk Enumeration (Перечисление рисков)

**Метод: STRIDE per Component**

Для каждого компонента системы задай:

```
S - Spoofing (Подмена личности)
   - Может ли кто-то выдать себя за пользователя/сервис?
   
T - Tampering (Изменение данных)
   - Может ли кто-то изменить данные в транзите/хранилище?
   
R - Repudiation (Отказ от действий)
   - Может ли кто-то отрицать совершенное действие?
   
I - Information Disclosure (Раскрытие информации)
   - Может ли кто-то получить доступ к закрытым данным?
   
D - Denial of Service (Отказ в обслуживании)
   - Может ли кто-то нарушить доступность системы?
   
E - Elevation of Privilege (Повышение привилегий)
   - Может ли обычный пользователь получить права админа?
```

**Критичность = Вероятность × Влияние**
- Вероятность: Низкая/Средняя/Высокая (на основе сложности эксплуатации)
- Влияние: Низкое/Среднее/Высокое (на основе ценности актива)

---

### **A** - Attack Surface Analysis (Анализ поверхности атаки)

**Вопросы:**

```
1. Публичные эндпоинты
   - Какие API endpoints доступны без аутентификации?
   - Какие требуют только basic auth?
   - Есть ли rate limiting?
   - Есть ли input validation?

2. Обработка пользовательского ввода
   - Где принимается пользовательский ввод?
   - Как валидируется?
   - Есть ли sanitization?
   - Используются ли prepared statements (SQL injection защита)?

3. Аутентификация и авторизация
   - Где хранятся токены/секреты?
   - Как проверяется валидность токенов?
   - Есть ли session fixation?
   - Как обрабатывается logout?

4. Конфигурация и секреты
   - Где хранятся секреты? (env vars, config files, vault?)
   - Есть ли hardcoded credentials?
   - Логируются ли секреты?
   - Есть ли default passwords?

5. Зависимости
   - Какие версии библиотек используются?
   - Есть ли известные CVE в зависимостях?
   - Обновляются ли зависимости регулярно?
   - Используются ли непроверенные библиотеки?

6. Сетевая безопасность
   - Какие порты открыты?
   - Используется ли TLS/encryption в transit?
   - Есть ли firewall rules?
   - Открыт ли debug/admin интерфейс публично?
```

**Действие:** Составь список всех векторов атаки с приоритетом (критичность × легкость эксплуатации).

---

### **P** - Path to Compromise (Путь к компрометации)

**Метод: Attack Trees**

Для каждой найденной уязвимости нарисуй дерево атаки:

```
КОРЕНЬ: Компрометация системы/актива

УЗЛЫ: Промежуточные шаги
  ├─ Шаг 1: Получить доступ к X
  │   ├─ Вариант A: Эксплуатировать уязвимость Y
  │   └─ Вариант B: Использовать социальную инженерию
  ├─ Шаг 2: Повысить привилегии
  └─ Шаг 3: Достичь цели

ЛИСТЬЯ: Конкретные техники (CWE, CAPEC, MITRE ATT&CK)
```

**Вопросы для каждого пути:**
```
1. Какие предварительные условия нужны?
2. Какие препятствия есть на пути?
3. Можно ли предотвратить на каждом шаге?
4. Какие логи/мониторинг зафиксируют попытку?
```

## 🔍 НАХОЖДЕНИЕ КОРНЕВОЙ ПРИЧИНЫ (Root Cause Analysis)

### Правило "5 Почему" + "Так что же?"

```
Проблема: SQL Injection в API endpoint

Почему 1: Пользовательский ввод попадает напрямую в SQL запрос
Почему 2: Нет использования prepared statements
Почему 3: Разработчик не знал о рисках
Почему 4: Нет code review процесса
Почему 5: Нет security guidelines в документации

"Так что же?" → КОРНЕВАЯ ПРИЧИНА:
  - Отсутствие security training для разработчиков
  - Отсутствие автоматизированных security checks в CI/CD
  - Отсутствие security review в процессе разработки
```

### Признаки симптома vs корневой причины:

**СИМПТОМ** (плохое решение):
- "Патч: добавить проверку только в этом месте"
- "Временный fix: валидация на фронтенде"
- "Workaround: blacklist определенных символов"

**КОРНЕВАЯ ПРИЧИНА** (правильное решение):
- "Архитектурное изменение: использовать ORM с автоматической защитой"
- "Процессное изменение: добавить security lint в CI/CD"
- "Обучение: провести security training по OWASP Top 10"

## 🛡️ ПРОЕКТИРОВАНИЕ РЕШЕНИЙ

### Принципы безопасного дизайна:

**1. Defense in Depth (Многослойная защита)**
```
❌ Плохо: Только валидация на фронтенде
✅ Хорошо: Валидация на фронтенде + валидация на бэкенде + валидация в БД
```

**2. Principle of Least Privilege (Минимальные привилегии)**
```
❌ Плохо: Все сервисы имеют доступ ко всей БД
✅ Хорошо: Каждый сервис имеет доступ только к нужным таблицам/полям
```

**3. Fail Secure (Безопасный отказ)**
```
❌ Плохо: При ошибке валидации - пропустить запрос
✅ Хорошо: При ошибке валидации - отклонить запрос
```

**4. Separation of Concerns (Разделение ответственности)**
```
❌ Плохо: Один сервис обрабатывает auth и бизнес-логику
✅ Хорошо: Отдельный auth-service + бизнес-сервисы
```

**5. Secure by Default (Безопасно по умолчанию)**
```
❌ Плохо: Все API открыты, нужно закрыть специально
✅ Хорошо: Все API закрыты по умолчанию, нужно открыть специально
```

### Структура решения:

```markdown
## Решение для [Проблема]

### 1. Немедленные меры (Mitigation)
   - Что: [конкретное действие]
   - Зачем: [предотвратить эксплойт прямо сейчас]
   - Риск: [если не сделать - что случится]

### 2. Краткосрочные меры (Patch)
   - Что: [конкретное исправление]
   - Зачем: [закрыть вектор атаки]
   - Таймлайн: [когда выполнить]

### 3. Долгосрочные меры (Fix)
   - Что: [архитектурное изменение]
   - Зачем: [предотвратить повторение]
   - Таймлайн: [когда выполнить]

### 4. Мониторинг и детекция
   - Что отслеживать: [метрики/логи/события]
   - Как детектировать: [правила/алерты]
   - Что делать при детекции: [response plan]

### 5. Валидация решения
   - Как проверить что fix работает: [тесты/чеклист]
   - Как убедиться что не сломал функциональность: [регрессионные тесты]
```

## 🎓 ПРАКТИЧЕСКИЕ ПРИМЕРЫ АНАЛИЗА

### Пример 1: Web API

**Система:** REST API для финансовых транзакций

```
S - System Map:
  Assets: Балансы пользователей, приватные ключи, история транзакций
  Threats: Финансовые мошенники, инсайдеры, хакеры
  Trust Boundaries: User → API → Database, API → Payment Gateway
  Data Flow: Request → Validation → Business Logic → Database → Response

R - Risk Enumeration:
  T - Tampering: Пользователь может изменить сумму транзакции в запросе [ВЫСОКИЙ]
  I - Information Disclosure: API возвращает внутренние ошибки с деталями [СРЕДНИЙ]
  E - Elevation: Можно ли перевести больше чем есть на балансе? [КРИТИЧНЫЙ]

A - Attack Surface:
  - POST /api/transfer - принимает сумму, нет проверки баланса
  - GET /api/user/{id}/balance - возвращает баланс любому, кто знает ID
  - Error responses содержат stack traces

P - Path to Compromise:
  1. Найти валидный user ID (через enumeration)
  2. Узнать баланс (через GET /api/user/{id}/balance)
  3. Отправить транзакцию на сумму больше баланса (эксплуатировать отсутствие проверки)
  
Root Cause:
  Симптом: Нет проверки баланса перед транзакцией
  Корневая причина: Отсутствие валидации бизнес-правил в service layer

Решение:
  1. Mitigation: Добавить проверку баланса СЕЙЧАС
  2. Patch: Реализовать атомарную транзакцию в БД
  3. Fix: Ввести domain events для всех финансовых операций + audit log
```

### Пример 2: Микросервисная архитектура

**Система:** E-commerce с микросервисами

```
Проблема: Service A вызывает Service B без проверки подлинности

Root Cause Analysis:
  Почему 1: Service-to-service communication без mTLS/auth
  Почему 2: Разработчики считали internal network безопасным
  Почему 3: Нет service mesh или API gateway для внутренних вызовов
  Почему 4: Архитектура создана без учета security requirements

"Так что же?" → КОРНЕВАЯ ПРИЧИНА:
  - Отсутствие security-first подхода при проектировании
  - Отсутствие zero-trust политики для внутренних сервисов

Решение:
  1. Mitigation: Добавить API keys для service-to-service (быстро)
  2. Patch: Внедрить mTLS между сервисами (medium term)
  3. Fix: Внедрить service mesh (Istio/Linkerd) с автоматическим mTLS (long term)
```

## 🧠 МЕНТАЛЬНЫЕ МОДЕЛИ ДЛЯ БЫСТРОГО АНАЛИЗА

### Модель 1: "Что если я злоумышленник?"

Для каждого компонента задай:
- "Что я могу украсть/повредить здесь?"
- "Что самое ценное в этом компоненте?"
- "Как я могу использовать это для доступа к другим компонентам?"

### Модель 2: "Trust Boundary Crossing"

Всякий раз когда данные пересекают границу доверия:
- **Всегда проверяй** подлинность и целостность
- **Всегда валидируй** формат и содержимое
- **Всегда логируй** пересечение границы
- **Всегда ограничивай** привилегии на новой стороне

### Модель 3: "Изменение состояния = риск"

Всякий раз когда что-то изменяется:
- **Проверь авторизацию** - может ли пользователь делать это?
- **Проверь бизнес-правила** - валидно ли это изменение?
- **Проверь атомарность** - не создает ли это race condition?
- **Проверь аудит** - логируется ли изменение?

## 📚 РЕСУРСЫ ДЛЯ ГЛУБОКОГО ПОНИМАНИЯ

### Framework и стандарты:
- **OWASP Top 10** - базовый список уязвимостей веб-приложений
- **CWE** - Common Weakness Enumeration (каталог уязвимостей)
- **CAPEC** - Common Attack Pattern Enumeration (каталог техник атак)
- **MITRE ATT&CK** - тактики и техники реальных атак
- **NIST Cybersecurity Framework** - структура управления рисками

### Методологии:
- **STRIDE** - модель угроз от Microsoft
- **PASTA** - Process for Attack Simulation and Threat Analysis
- **DREAD** - система оценки рисков
- **CVSS** - система оценки критичности уязвимостей

### Практика:
- **HackTheBox** - практика эксплуатации
- **PortSwigger Web Security Academy** - обучение веб-безопасности
- **CTF (Capture The Flag)** - соревнования по безопасности

## 🎯 КАК ИСПОЛЬЗОВАТЬ ЭТОТ ПРОМПТ

### Сценарий 1: Анализ существующей системы

```
Шаг 1: Используй S.R.A.P. методологию
Шаг 2: Заполни каждый раздел подробно
Шаг 3: Найди 3-5 критичных проблем
Шаг 4: Для каждой проведи Root Cause Analysis
Шаг 5: Спроектируй решения используя структуру "5 типов мер"
```

### Сценарий 2: Code Review с фокусом на безопасность

```
Для каждого PR:
1. Найди все места обработки пользовательского ввода
2. Проверь все места изменения состояния (DB writes, file writes)
3. Проверь все места взаимодействия с внешними сервисами
4. Используй чеклист из раздела "Attack Surface Analysis"
```

### Сценарий 3: Проектирование новой системы

```
1. Начни с System Map (определи assets, threats, boundaries)
2. Используй STRIDE для каждого компонента на этапе дизайна
3. Закладывай принципы безопасного дизайна с начала
4. Определи security requirements до начала разработки
```

## 💡 ЗОЛОТЫЕ ПРАВИЛА БЕЗОПАСНОСТИ

1. **"Не доверяй никому"** - валидируй все на каждом слое
2. **"Не изобретай велосипед"** - используй проверенные библиотеки и паттерны
3. **"Предполагай компрометацию"** - проектируй с учетом что часть системы может быть скомпрометирована
4. **"Безопасность != Удобство"** - иногда нужно жертвовать UX ради безопасности
5. **"Логируй всё критичное"** - без логов невозможно понять что случилось
6. **"Тестируй безопасность"** - security testing должен быть частью QA процесса
7. **"Обновляйся регулярно"** - устаревшие зависимости = известные уязвимости
8. **"Минимум привилегий"** - каждый компонент должен иметь только необходимые права

---

**Помни:** Безопасность - это процесс, а не состояние. Система никогда не будет "полностью безопасной", но может быть "безопасной достаточно" для текущих угроз и рисков.
