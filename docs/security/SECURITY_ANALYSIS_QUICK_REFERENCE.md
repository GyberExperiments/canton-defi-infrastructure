# ⚡ Быстрая шпаргалка: Анализ безопасности за 30 минут

## 🔥 S.R.A.P. - Мини-версия

### S - System Map (5 минут)
```
□ Что защищаем? (данные, деньги, репутация)
□ Кто может атаковать? (внутренние/внешние)
□ Где границы доверия? (User→API→DB→External)
□ Как данные текут? (source → process → store → transmit)
```

### R - Risk Enumeration (10 минут)
```
Для каждого компонента проверь STRIDE:
□ Spoofing - подмена пользователя/сервиса?
□ Tampering - изменение данных?
□ Repudiation - отрицание действий?
□ Information Disclosure - утечка данных?
□ Denial of Service - нарушение доступности?
□ Elevation - повышение привилегий?
```

### A - Attack Surface (10 минут)
```
□ Публичные API без auth/rate limiting?
□ User input без валидации/sanitization?
□ Секреты в коде/logs/config?
□ Устаревшие зависимости с CVE?
□ Открытые debug/admin интерфейсы?
□ Слабая/отсутствующая авторизация?
```

### P - Path to Compromise (5 минут)
```
Для каждой уязвимости:
□ Какие шаги нужны для эксплуатации?
□ Какие препятствия есть на пути?
□ Можно ли предотвратить на каждом шаге?
□ Что залогирует попытку?
```

## 🛡️ Быстрые решения (по принципам)

```
❌ Плохо → ✅ Хорошо

1. Только frontend validation
   → Frontend + Backend + DB constraints

2. Все API открыты
   → Все закрыты по умолчанию, открыть явно

3. Один сервис = все права
   → Минимальные привилегии на компонент

4. При ошибке = пропустить
   → При ошибке = отклонить (fail secure)

5. Логи содержат секреты
   → Sanitize логи перед записью
```

## 📋 Чеклист критичных проверок

### API Security
- [ ] Все эндпоинты требуют auth (кроме публичных)
- [ ] Rate limiting на все публичные эндпоинты
- [ ] Валидация всех входных данных
- [ ] Prepared statements (SQL injection защита)
- [ ] Error handling не раскрывает внутренности
- [ ] HTTPS везде (no HTTP)

### Authentication & Authorization
- [ ] Токены имеют expiry
- [ ] Пароли хешируются (bcrypt/argon2)
- [ ] Session management (CSRF protection)
- [ ] Проверка прав на каждое действие
- [ ] Logout инвалидирует сессию

### Data Security
- [ ] Секреты в vault/env (не в коде)
- [ ] Encryption at rest для чувствительных данных
- [ ] Encryption in transit (TLS)
- [ ] PII/PCI данные замаскированы в логах

### Dependencies
- [ ] Нет известных CVE (`npm audit`, `snyk test`)
- [ ] Зависимости обновляются регулярно
- [ ] Нет подозрительных/непроверенных пакетов

### Infrastructure
- [ ] Нет default credentials
- [ ] Закрыты ненужные порты
- [ ] Firewall rules настроены
- [ ] Мониторинг и alerting активны

## 🎯 Типичные проблемы и корневые причины

| Симптом | Корневая причина | Решение |
|---------|------------------|---------|
| SQL Injection | Нет prepared statements | ORM/query builder с параметризацией |
| XSS | Нет sanitization | Content Security Policy + sanitize input |
| Broken Auth | Слабая session management | Secure token handling + refresh tokens |
| Sensitive Data Exposure | Логи содержат секреты | Sanitize logs + structured logging |
| XXE | Парсинг XML без защиты | Disable XXE в парсере |
| Broken Access Control | Нет проверки прав | Middleware для проверки на каждом эндпоинте |
| Security Misconfiguration | Default passwords/настройки | Security hardening checklist |
| Insecure Deserialization | Небезопасный парсинг | Use safe serialization libraries |
| Insufficient Logging | Нет audit trail | Structured logging + SIEM |
| SSRF | Прямой вызов URL из input | Whitelist разрешенных доменов |

## 💡 Быстрые вопросы для любой системы

1. **"Что самое ценное здесь?"** → Это нужно защищать в первую очередь
2. **"Что произойдет если это сломается?"** → Это критичный компонент
3. **"Кто может это сделать?"** → Проверь авторизацию
4. **"Как я узнаю об атаке?"** → Нужен мониторинг/логирование
5. **"Что если часть системы скомпрометирована?"** → Нужна изоляция

## 🚨 Red Flags (немедленно проверить)

- 🔴 Пользовательский ввод попадает в SQL/команды/файлы без валидации
- 🔴 Секреты в git history/public repos
- 🔴 API без rate limiting
- 🔴 Пароли в plain text или слабое хеширование
- 🔴 Debug mode в production
- 🔴 Публичный доступ к admin панелям
- 🔴 Отсутствие CORS политик
- 🔴 Error messages с stack traces
- 🔴 Отсутствие CSRF protection
- 🔴 Прямой доступ к БД из фронтенда

```

  Высокое влияние × Легкая эксплуатация

СРЕДНИЙ ПРИОРИТЕТ (исправить в этом месяце):
  Среднее влияние × Средняя/Сложная эксплуатация
  Низкое влияние × Легкая эксплуатация

НИЗКИЙ ПРИОРИТЕТ (планировать):
  Низкое влияние × Сложная эксплуатация
```
