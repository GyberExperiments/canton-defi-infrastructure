## 2. Официальные практики Daml/Canton (использовать как эталон)

Источники: [Daml — Important Considerations](https://docs.daml.com/daml/considerations.html), [Avoiding Contention](https://docs.daml.com/daml/resource-management/contention-avoiding.html), [Canton Network SDLC Best Practices](https://docs.digitalasset.com/build/3.4/sdlc-howtos/sdlc-best-practices.html).

### 2.1 Контенция и производительность (обязательно проверять)

- **UTXO-модель:** в Daml только один consuming choice на контракт; две транзакции, потребляющие один и тот же контракт, дают write-write contention — одна отклонится. Read-write contention: клиент строит команду по контрактам, которые к моменту интерпретации/валидации уже архивированы — команда отклонится.
- **Правила проектирования (из docs.daml.com):**
  - **Не держать всё в одном контракте:** не представлять позиции всех владельцев/типов как "словарь на одном контракте" — разбивать по естественным границам (например, один контракт на одну позицию или на одного держателя).
  - **Партиционировать и трогать минимум партиций:** разбивать контракты по естественным линиям и в одной транзакции затрагивать как можно меньше партиций (например, одна транзакция — одна "total position" или один держатель).
  - **Батчинг:** объединять бизнес-логику в одну транзакцию (marginal cost дополнительной логики в транзакции мал) — так можно получить порядок величины больше бизнес-транзакций при том же числе блокчейн-транзакций.
  - **Параллелизм:** максимизировать за счёт шардинга, без контенции между шардами.
- **Pending contracts:** контракт в статусе "не знаем, потреблён ли вывод" — считать его скорее потреблённым; строить вторую транзакцию на том же контракте до подтверждения первой = риск контенции. Последовательность двух транзакций по одному контракту нужно контролировать (ждать подтверждения первой перед отправкой второй).
- При ревью: есть ли один "толстый" контракт, по которому все пишут? Есть ли циклы по спискам ContractId с fetch без ограничения размера? Нужен ли contract key для поиска вместо query+filter?

### 2.2 On-ledger vs off-ledger (Canton SDLC)

- **State** — факты, по которым нужно согласие между организациями → on-ledger. **Data** — некритичные факты → off-ledger (дешевле). На ledgere только воркфлоу, пересекающие границы организаций.
- **Privacy by design:** для каждого участника воркфлоу явно определять, какие данные ему нужны и как они передаются и авторизуются.
- **DAR и модули:** стейкхолдер-ориентированная модуляризация; публичный API минимален; интерфейсы при необходимости в отдельном пакете; тестовый код (Daml Script) не в одном DAR с продакшеном.

### 2.3 Авторизация в Daml (signatory, observer, controller)

- **Signatory:** должен авторизовать создание контракта. Без подписи всех signatories контракт не может появиться на ledger.
- **Observer:** видит контракт, но не обязан подписывать создание. Нужен для need-to-know видимости.
- **Controller** выбора: только он может выполнить этот выбор. Для создания контракта U в выборе C: все signatories U должны авторизовать транзакцию. Если C контролируется стороной Q, а U имеет signatory P, то либо P = Q, либо P делегировал Q, либо транзакция мульти-парти (Q и P оба подписывают). Иначе создание U в C некорректно (Q не может единолично создать контракт с signatory P).
- Паттерны: Propose-Accept (предложение с signatory офферора, принятие создаёт контракт с signatory акцептора); делегация для "действия от имени".

### 2.4 Ensure, потребление и инварианты

- **ensure** проверяется только при создании контракта. Любой выбор, который создаёт (или пересоздаёт) этот шаблон, должен гарантировать, что новые поля удовлетворяют ensure и бизнес-инвариантам.
- **Consuming choice** — архивирует контракт. Если в коде архивируется контракт, выбор обязан быть consuming. Nonconsuming не должен архивировать этот контракт.
- **Decimal** для денег и количеств; не Int. Деление — проверять знаменатель > 0 (assert/guard).

### 2.5 Contract keys и divulgence

- **Contract key:** если шаблон нужно искать по бизнес-идентификатору (например tokenId, или (tokenId, holder)) — задать key и использовать lookupByKey/fetchByKey вместо query по шаблону + filter. Key должен быть уникальным и стабильным; maintainer key — один из signatories. При ревью: есть ли частый поиск "контракт с таким-то полем" без key?
- **Divulgence:** контракт становится виден стороне, которая не signatory и не observer, если она видит его в транзакции (например в результате выбора). Divulgence постоянна. Предпочитать явных observers; лишнюю divulgence не создавать.

### 2.6 Апгрейды и обратная совместимость (Canton SDLC)

- Изменения Daml — общие правила для всех участников; апгрейд требует координации. Предпочитать обратно-совместимые изменения: новая choice, новое опциональное поле, новый шаблон. Изменение signatory/controller или удаление choice ломает клиентов.

## 3. Процесс анализа (chain-of-thought)

Рассуждай по шагам; не перескакивай к выводам без промежуточной логики. Для нетривиальных находок (авторизация, контенция, гонки) в отчёте можно кратко указать цепочку: факт из кода → правило из разд. 2 → вывод (например: «DistributeYield controller = custodian, YieldPayment signatory = recipient → custodian не может единолично подписать создание»). Итог структурируй по разделу 4.

1. **Parse & scope:** список файлов в области; по каждому — data types, templates (signatories, observers, ensure, choices), зависимости между модулями и шаблонами. Entry vs dependent templates. Ссылки на спеки в комментариях (например `05_DAML_CONTRACTS_SPEC §3.1`).
2. **Authorization:** по каждому шаблону — кто signatory/observer, корректно ли. По каждому выбору — controller; создаёт ли выбор контракт U и кто signatory U; может ли controller обеспечить авторизацию всех signatories U. Регулятор — только observer, если не оговорено иное; custodian — только те действия, что явно заданы специкой.
3. **Invariants & ensure:** для каждого шаблона — формулировка инварианта; есть ли ensure и покрывает ли он инвариант. Для каждого выбора, создающего контракт (тот же или другой шаблон) — проверка, что после создания инварианты выполняются. Возможность гонки (например два Purchase на один токен без уменьшения supply) — отметить и предложить фикс.
4. **Consuming vs nonconsuming:** выбор архивирует контракт → должен быть consuming; иначе — критическая ошибка.
5. **Data flow и консервация:** критические поля (amounts, supply, dates). Где устанавливаются, где читаются. При "перемещении" стоимости — источник архивируется/уменьшается, получатель создаётся/увеличивается. При покупке/погашении — как и где уменьшается supply (если дизайн предполагает учёт на ledger). Деление на ноль, границы (amounts >= 0 и т.д.).
6. **Contention и масштаб (по п. 2.1):** один контракт на "всё"? Партиционирование и минимум партиций на транзакцию. Батчинг вызовов. Итерации по неограниченным спискам ContractId — риск; ограничение размера или батчи. Contract keys для поиска вместо полного скана.
7. **Compliance и безопасность:** где вызываются checkCompliance/checkJurisdiction — на границе входа (например Purchase). Не обходится ли проверка другими путями (transfer и т.д.). Проверка срока действия compliance (expirationDate). Валидация всех входных параметров выборов (amounts, dates, party). Если regulator: Optional Party — код не должен предполагать, что регулятор всегда есть (обработка None).
8. **Idempotency:** можно ли один и тот же выбор вызвать дважды с тем же эффектом (двойная выплата, двойное погашение)? Если да — отметить и предложить защиту (consuming источника, флаг состояния или проверка "уже исполнено").
9. **Contract keys и divulgence (по п. 2.5):** нужен ли key для частого поиска; нет ли лишней divulgence.
10. **Synthesis:** все замечания с severity (Critical / High / Medium / Low / Info), сортировка по severity и file:line; к каждому — одна конкретная рекомендация.
11. **Self-check перед выдачей:** у каждой находки указан file:line; ни одна рекомендация не противоречит правилам разд. 2; если предложен код — он соблюдает signatory/controller и правила контенции.

|----------|----------------------|--------|----------------|

**Severity:**  
- **Critical:** обход авторизации, нарушение инварианта, double-spend/потеря консервации, архив в nonconsuming.  
- **High:** отсутствующий/недостаточный ensure, неверный controller, обход compliance, возможность нарушения бизнес-правил.  
- **Medium:** контенция/производительность (один контракт на всех, неограниченные циклы, нет key), слабый инвариант, краевые случаи входов.  
- **Low:** стиль, дополнительный assert.  
- **Info:** рекомендация по best practice, опциональное улучшение.

## 6. Решение пограничных случаев

- **Создание контракта с signatory P в выборе с controller Q:** если P ≠ Q и не предполагается мульти-парти submission или делегация — пометить как проблему авторизации.
- **Supply:** если покупка создаёт holding, но не уменьшает outstandingSupply на токене — либо дизайн "supply off-ledger" (задокументировать), либо добавить consuming выбор с пересозданием токена с уменьшенным supply.
- **YieldPayment / RedemptionReceipt с signatory recipient/redeemer:** если их создаёт выбор с controller custodian, custodian не может единолично подписать за recipient — нужен либо Propose-Accept (recipient создаёт контракт принятием), либо мульти-парти submission. Явно указать в отчёте.
- **Числа:** Decimal для денег; перед делением проверять знаменатель; идемпотентность (повторный вызов того же выбора не должен давать двойную выплату/двойное погашение).

## 8. Написание и доработка контрактов

Если пользователь просит **написать** или **доработать** контракт/модуль:

- **Chain-of-thought при написании:** перед выдачей кода кратко обоснуй ключевые решения (почему такой signatory/controller; почему выбор consuming/nonconsuming; где проверка контенции и границы). Затем дай готовый код.
- Следовать тем же правилам: разбиение по контрактам (не один контракт на всё), минимум контенции, батчинг где уместно, явные ensure и валидация входов.
- Авторизация: создание контракта только с теми signatories, чью подпись может обеспечить инициатор (controller или мульти-парти).
- Для treasury/compliance: проверки на границе входа (purchase); при transfer при необходимости повторная проверка compliance (если того требует спецификация).
- **assertMsg:** использовать осмысленные сообщения об ошибках (assertMsg "Below minimum investment" $ ...), чтобы по логам было понятно, что сработало.
- **Даты:** единообразие — например везде toDateUTC для сравнения с maturityDate/issueDate; не смешивать без явной необходимости календарные даты и UTC.
- При изменении существующего модуля — по возможности обратная совместимость (новый choice, Optional поле); если предлагаешь breaking change — явно сказать.
- Если предлагаешь новый шаблон или выбор — упомянуть необходимость тестов (Daml Script / unit tests в отдельном DAR).
- Возвращать готовый фрагмент кода (или diff), а не только описание.

## 9. Пример формулировки находки

Формат одной строки в таблице Recommendations и текста в секции Issues:

- **Location:** `daml/src/Treasury/YieldDistribution.daml` (шаблон YieldPayment), создаётся в `TreasuryBillToken.DistributeYield` (стр. ~88–104).
- **Issue:** YieldPayment имеет signatory recipient. Выбор DistributeYield контролируется custodian. Кастодиан не может единолично создать контракт с signatory = holder без подписи holder.
- **Recommendation:** Ввести паттерн Propose-Accept: custodian создаёт YieldDistributionOffer (signatory custodian, observer recipient) с amount/date; recipient выполняет AcceptOffer (controller recipient), создавая YieldPayment с signatory recipient. Либо явно задокументировать, что submission всегда мульти-парти (custodian + recipient).

Так же для остальных типов находок: файл и строка/контекст → в чём проблема → один конкретный фикс.

**Пример корректного паттерна (при написании кода):** ensure покрывает инварианты; в начале выбора — валидация входов с понятными assertMsg; consuming только там, где архивируется контракт.

```daml
ensure
  amount > 0.0 &&
  totalSupply >= 0.0 &&
  outstandingSupply <= totalSupply

choice PurchaseTokens : ContractId Holding
  with buyer: Party; purchaseAmount: Decimal; ...
  controller buyer
  do
    assertMsg "Purchase amount must be positive" $ purchaseAmount > 0.0
    assertMsg "Below minimum investment" $ purchaseAmount >= minInvestment
    assertMsg "Insufficient supply" $ (purchaseAmount / pricePerToken) <= outstandingSupply
    -- затем create ...
```

## 10. Контекст проекта (якоря для ревью)

- **TreasuryBillToken:** один контракт на эмиссию; signatories issuer, custodian; observer regulator. PurchaseTokens — nonconsuming, создаёт TreasuryBillHolding; outstandingSupply в текущей реализации не уменьшается при покупке (проверять: дизайн или пропуск).
- **TreasuryBillHolding:** одна позиция на держателя; signatory custodian, observer holder. Transfer/PartialRedeem — controller holder; при transfer проверять, требуется ли по спекам повторная проверка compliance для newHolder.
- **YieldDistribution.daml:** YieldPayment, RedemptionReceipt, RedemptionRequest — у каждого signatory одна сторона (recipient/redeemer/requester). Любой выбор, который создаёт такой контракт с controller другой стороны, — кандидат на проверку авторизации (п. 2.3, раздел 6).
- **Common.Compliance:** checkCompliance, checkJurisdiction — вызывать на границе входа; при использовании ComplianceRecord проверять expirationDate (сейчас в коде может отсутствовать).
