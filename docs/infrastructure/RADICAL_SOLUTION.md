# 🔥 РАДИКАЛЬНЫЕ РЕШЕНИЯ ДЛЯ КРИТИЧЕСКОЙ СИТУАЦИИ

## ПРОБЛЕМА
- API server перегружен на 99% CPU
- Тысячи Pending подов блокируют scheduler
- Обычные команды kubectl таймаутят
- Очистка идет очень медленно или не работает

## САМОЕ ДЕЙСТВЕННОЕ РЕШЕНИЕ

### 🎯 РЕКОМЕНДУЕМЫЙ ПОДХОД: `radical-cleanup-final.sh`

**Почему это самое эффективное:**
1. **Параллельное выполнение** - использует `xargs -P 50-100` для параллельного удаления
2. **Массовые операции** - удаляет сотни ресурсов одновременно
3. **Обход finalizers** - очищает зависшие поды через patch
4. **Увеличенные лимиты** - `ulimit -n 4096` для большего количества одновременных операций
5. **Короткие таймауты** - `--timeout=1-2s` чтобы не ждать зависшие операции

**Использование:**
```bash
./scripts/radical-cleanup-final.sh
```

**Что делает:**
1. Удаляет ВСЕ Challenge, CertificateRequest, Order ресурсы параллельно
2. Удаляет Pending поды из всех проблемных namespace параллельно
3. Удаляет все cert-manager challenge поды по паттерну имени
4. Очищает finalizers у всех Terminating подов
5. Финальная очистка всех оставшихся Pending подов

### 🔥 АЛЬТЕРНАТИВА: `radical-cleanup.sh`

Более безопасная версия с меньшей параллельностью:
```bash
./scripts/radical-cleanup.sh
```

### ⚠️ КРИТИЧЕСКИЙ СЛУЧАЙ: `radical-cleanup-etcd.sh`

Только если API server полностью недоступен:
```bash
./scripts/radical-cleanup-etcd.sh
```

## ТЕХНИЧЕСКИЕ ДЕТАЛИ

### Почему это работает:

1. **Параллельность (`xargs -P`)**
   - Обычные команды выполняются последовательно
   - `xargs -P 50-100` выполняет 50-100 операций одновременно
   - Это в 50-100 раз быстрее

2. **Короткие таймауты**
   - `--timeout=1-2s` вместо дефолтных 30s
   - Не ждет зависшие операции
   - Продолжает с другими

3. **Массовые операции**
   - Удаляет сотни ресурсов за раз
   - Использует `head -1000` для ограничения батча
   - Обрабатывает все namespace параллельно

4. **Обход finalizers**
   - Зависшие поды часто имеют finalizers
   - `kubectl patch` очищает их напрямую
   - Позволяет удалить поды, которые "зависли"

5. **Увеличенные лимиты**
   - `ulimit -n 4096` увеличивает лимит открытых файлов
   - Необходимо для большого количества параллельных операций

## СРАВНЕНИЕ ПОДХОДОВ

| Подход | Скорость | Безопасность | Когда использовать |
|--------|----------|--------------|-------------------|
| Обычные команды | Медленно | Безопасно | Нормальная ситуация |
| `radical-cleanup.sh` | Быстро | Безопасно | Много Pending подов |
| `radical-cleanup-final.sh` | Очень быстро | Безопасно | Критическая ситуация |
| `radical-cleanup-etcd.sh` | Мгновенно | Опасно | API server недоступен |

## РЕКОМЕНДАЦИИ

1. **Начать с `radical-cleanup-final.sh`** - самое эффективное решение
2. **Проверить результат** - посмотреть статистику после выполнения
3. **Повторить если нужно** - скрипт идемпотентный, можно запускать несколько раз
4. **Мониторить нагрузку** - после очистки проверить `kubectl top node tmedm`

## ОЖИДАЕМЫЙ РЕЗУЛЬТАТ

- ✅ Challenge ресурсов: < 10 (было сотни)
- ✅ Нагрузка на control-plane: < 50% CPU (было 99%)
