## 🔍 ОБНАРУЖЕННЫЕ ПРОБЛЕМЫ В ХОДЕ РАБОТЫ

### 1. ПЕРЕГРУЗКА CONTROL-PLANE УЗЛА (tmedm)
**Симптомы:**
- CPU загрузка стабильно **99%** (3998-3999m из ~4000m)
- Memory: 64-67%
- Узел выполняет роль control-plane и master

**Наблюдения:**
- Системные поды (coredns, metrics-server) потребляют мало CPU (<10m каждый)
- Проблема НЕ в подах, а в самом control-plane процессе (k3s server)
- API server отвечает с огромными задержками (40-45 секунд на простые запросы)

**Гипотеза:** API server перегружен обработкой огромного количества объектов (тысячи Pending подов)

### 2. МАССИВНОЕ КОЛИЧЕСТВО PENDING ПОДОВ
**Детали:**
- **~6,200-7,000 Pending подов** только в `platform-gyber-org`
- Все поды имеют префикс `cm-acme-http-solver-*` (cert-manager challenges)
- Возраст подов: от 13 часов до 2+ дней
- Все поды в статусе `Pending` с `READY 0/1`

**Проблемы при удалении:**
- `kubectl get pods` с field-selector зависает на 40-45 секунд
- `kubectl delete pod` занимает 5-37 секунд на каждый под
- Большинство операций удаления завершаются ошибками таймаута
- API server не может обработать запросы из-за перегрузки

**Наблюдения:**
- Удаление по 10 подов работает, но очень медленно
- Каждая операция удаления занимает 5-37 секунд
- Получение списка подов занимает 40-45 секунд

### 3. ПРОБЛЕМЫ С CERT-MANAGER
**Обнаружено:**
- Challenge в статусе `pending`: `platform-gyber-org-tls-3-2000845828-2697340219`
- Причина: `Timeout: request did not complete within requested timeout - context deadline exceeded`
- Сертификат `platform-gyber-org-tls` в статусе `Ready: True`
- Но есть условие: `Renewing certificate as renewal was scheduled at 2026-01-10 08:52:44`

**Проблема:**
- Cert-manager пытается обновить сертификат
- Challenge не может завершиться из-за таймаутов
- Cert-manager продолжает создавать новые challenge поды
- Старые поды не удаляются, накапливаются

**Попытки исправления:**
- Удаление challenge через `kubectl delete challenge` работает, но challenge создается заново
- Namespace `cert-manager` пуст (нет deployments) - возможно, cert-manager работает в другом namespace

### 4. ЗАВИСШИЕ ПОДЫ НА ОТКЛЮЧЕННОМ УЗЛЕ
**Статус:**
- Узел `canton-node-65-108-15-30` в статусе `Ready,SchedulingDisabled`
- ~20-40 подов зависли в статусе `Terminating`
- Принудительное удаление через patch finalizers частично работает
