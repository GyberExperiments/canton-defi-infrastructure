# 🧪 Тестовый сценарий для валидации Canton адресов

## Цель теста
Убедиться, что все форматы Canton адресов проходят валидацию в UI и API.

## Тестовые адреса

### ✅ Валидные адреса (должны приниматься)

#### 1. Namespace формат (bron::)
```
bron::122064322dfc38d58c6b0d4e5bb77d7ff91e20bbd9321c3dc4
```
**Описание**: Адрес из скриншота пользователя - короткий вариант  
**Формат**: `namespace::fingerprint` с двойным двоеточием

#### 2. Namespace формат - полный
```
bron::1220dcd32eb92dd21a7418cfd3fe9f5bcaa3d88ced980004361ac57ca63db9f6275f
```
**Описание**: Полный namespace адрес  
**Формат**: `namespace::fingerprint` с длинным hex

#### 3. Альтернативный namespace (canton::)
```
canton::1220abcdef1234567890abcdef1234567890abcdef1234567890abcdef1234567890
```
**Описание**: Canton namespace вместо bron  
**Формат**: Любой namespace с :: и hex

#### 4. Классический формат
```
alice:1234567890abcdef1234567890abcdef12345678
```
**Описание**: Формат с одинарным двоеточием  
**Формат**: `name:fingerprint` (одно двоеточие)

#### 5. Hex-only формат
```
1220dcd32eb92dd21a7418cfd3fe9f5bcaa3d88ced980004361ac57ca63db9f6275f
```
**Описание**: Чистый hex fingerprint  
**Формат**: 32-80 hex символов без префикса

#### 6. Короткий hex
```
abcdef1234567890abcdef1234567890
```
**Описание**: Минимальная длина hex (32 символа)  
**Формат**: Ровно 32 hex символа

### ❌ Невалидные адреса (должны отклоняться)

#### 1. Слишком короткий
```
bron::1234
```
**Причина**: Fingerprint менее 32 символов

#### 2. Неправильные символы
```
test::xyz123invalid
```
**Причина**: Нон-hex символы в fingerprint

#### 3. Одно двоеточие с коротким fingerprint
```
bron:1234
```
**Причина**: Не соответствует ни одному формату

#### 4. Пустой адрес
```
(оставить поле пустым)
```
**Причина**: Обязательное поле

#### 5. Слишком короткий hex
```
123abc
```
**Причина**: Менее 32 символов

## Тестирование UI

### Шаг 1: Форма Wallet Details
1. Откройте страницу обмена: `http://localhost:3000`
2. Заполните первый шаг (выбор токенов и суммы)
3. Нажмите "Continue"

### Шаг 2: Тест Receiving Address
Для каждого **валидного** адреса:
1. Вставьте адрес в поле "Receiving Address"
2. **Ожидаемый результат**: 
   - ✅ Зеленая галочка появляется
   - ✅ Сообщение "Valid CC address"
   - ✅ Нет красной ошибки

Для каждого **невалидного** адреса:
1. Вставьте адрес в поле "Receiving Address"
2. **Ожидаемый результат**:
   - ❌ Красный восклицательный знак
   - ❌ Сообщение "Invalid Canton address format"
   - ❌ Зеленой галочки нет

### Шаг 3: Тест Refund Address
Refund Address должен принимать:
- ✅ Любой валидный Canton адрес
- ✅ TRON адрес (T + 33 символа)
- ✅ Ethereum адрес (0x + 40 hex)
- ✅ Пустое значение (опциональное поле)

## Тестирование API

### Тест через curl

#### Валидный запрос (namespace формат)
```bash
curl -X POST http://localhost:3000/api/create-order \
  -H "Content-Type: application/json" \
  -d '{
    "cantonAddress": "bron::122064322dfc38d58c6b0d4e5bb77d7ff91e20bbd9321c3dc4",
    "email": "test@example.com",
    "paymentAmount": 100,
    "paymentToken": "USDT_TRC20"
  }'
```
**Ожидаемый результат**: HTTP 200, заказ создан

#### Валидный запрос (hex-only формат)
```bash
curl -X POST http://localhost:3000/api/create-order \
  -H "Content-Type: application/json" \
  -d '{
    "cantonAddress": "1220dcd32eb92dd21a7418cfd3fe9f5bcaa3d88ced980004361ac57ca63db9f6275f",
    "email": "test@example.com",
    "paymentAmount": 100,
    "paymentToken": "USDT_TRC20"
  }'
```
**Ожидаемый результат**: HTTP 200, заказ создан

#### Невалидный запрос
```bash
curl -X POST http://localhost:3000/api/create-order \
  -H "Content-Type: application/json" \
  -d '{
    "cantonAddress": "invalid_address",
    "email": "test@example.com",
    "paymentAmount": 100,
    "paymentToken": "USDT_TRC20"
  }'
```
**Ожидаемый результат**: HTTP 400, ошибка валидации

## Возможные проблемы

### Проблема: Валидация проходит в тестах, но не в UI
**Решение**: Проверить:
- Используется ли правильная функция `validateCantonAddress` из `@/lib/utils`
- Нет ли кеширования старого бандла (Ctrl+Shift+R)
- Правильно ли импортируются функции

### Проблема: Адрес принимается в UI, но отклоняется API
**Решение**: Проверить:
- Синхронизированы ли валидаторы в `cantonValidation.ts` и `utils.ts`
- Нет ли дополнительной валидации в middleware

### Проблема: Real-time валидация работает медленно
**Решение**: 
- Добавить debounce если необходимо
- Проверить производительность regex
