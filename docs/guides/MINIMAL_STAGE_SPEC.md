# Canton OTC — Minimal Stage: Исчерпывающее описание логики и задачи

Версия: 2025-10-17

## 1. Цель минимальной версии
Минимально‑жизнеспособная версия (minimal-stage) предназначена для безопасного ручного OTC‑обмена между клиентом и оператором. Сайт собирает заявку клиента, проводит валидации, создаёт запись в Google Sheets, уведомляет операторов в Telegram и предоставляет админ‑панель для ручной обработки заказа. Интеграции с блокчейнами сведены к валидации адресов и отображению реквизитов. Все перечисления происходят вручную оператором.

## 2. Основные роли
- Клиент: создаёт заявку на покупку/продажу, общается через Intercom.
- Оператор: обрабатывает заявку вручную по данным из Google Sheets/Telegram/админки.
- Администратор: управляет ценами/скидками/лимитами, контролирует статусы, мониторит здоровье сервиса.

## 3. Пользовательские потоки (E2E)
1) Создание заявки
   - Клиент заполняет форму (направление, сеть/токен, суммы, контактные данные).
   - Frontend проводит синхронные валидации (диапазоны, форматы, сети) и отправляет POST `/api/create-order`.
   - Backend:
     - Проводит строгую серверную валидацию, анти‑спам/рейт‑лимит.
     - Создаёт запись в Google Sheets (зафиксированные поля, статус `deposit`).
     - Отправляет уведомление в Telegram (операторская группа/канал).
     - Возвращает клиенту номер заявки и инструкции по дальнейшим шагам.

2) Отслеживание статуса
   - Клиент видит «прогресс‑бар» 5 стадий: `deposit → confirmation → exchange → sending → completed`.
   - Периодически опрашивает `/api/order-status?orderId=...`.

3) Ручная обработка оператором
   - Оператор видит новую строку в Google Sheets и/или нотификацию Telegram.
   - По готовности переводит статус строки/заказа (через админку или напрямую в таблице, если поддерживается).
   - Клиент видит обновление на сайте при следующем запросе статуса.

## 4. Админ‑панель (минимум)
- Аутентификация NextAuth v5 (локальные админ‑учётки с bcrypt‑хешем).
- Разделы: Dashboard (краткая статистика), Orders (список/фильтры/CRUD), Customers, Settings (цены/скидки/лимиты/рабочие часы), Logs/Health.

## 5. Архитектура (минимум)
- Next.js 15 (standalone), Node.js 20.
- API‑роуты: `/api/create-order`, `/api/order-status`, `/api/admin/*`, `/api/health`.
- Интеграции: Google Sheets (основной источник регистраций), Telegram Bot (уведомления), Intercom (коммуникация клиент↔оператор).
- Redis опционально (кэш/рейт‑лимит). Все внешние ключи в Kubernetes Secrets.

## 6. Конфигурация окружения (ключевые переменные)
- Google: `GOOGLE_SHEET_ID`, `GOOGLE_SERVICE_ACCOUNT_EMAIL`, `GOOGLE_PRIVATE_KEY`.
- Telegram: `TELEGRAM_BOT_TOKEN`, `TELEGRAM_CHAT_ID`, `TELEGRAM_BOT_USERNAME`.
- Цены/лимиты: `CANTON_COIN_PRICE_USD`, `CANTON_COIN_BUY_PRICE_USD`, `CANTON_COIN_SELL_PRICE_USD`, `MIN_USDT_AMOUNT`.
- Auth/Admin: `ADMIN_EMAIL`, `ADMIN_PASSWORD_HASH`, `ADMIN_API_KEY`, `NEXTAUTH_SECRET`, `NEXTAUTH_URL`.
- Публичные: `NEXT_PUBLIC_INTERCOM_APP_ID` (если Intercom включён), `NEXT_PUBLIC_APP_URL`.

## 7. API‑контракты (минимум)
- POST `/api/create-order`
  - Вход: направление, сеть/токен, суммы, контакты, согласия.
  - Выход: `orderId`, инструкции, `nextStatus` = `deposit`.
  - Ошибки: 400 (валидация), 429 (рейт‑лимит), 500 (интеграции).
- GET `/api/order-status?orderId=...`
  - Выход: `{ status, steps[], updatedAt }`.
- GET `/api/health`
  - Выход: статусы интеграций (без секретов), `build/version`, `uptime`.

## 8. Безопасность
- Rate limiting (по IP/заказам), анти‑спам (базовые эвристики).
- Строгая серверная валидация; защита от XSS/CSRF (формы неавторизованные, но входные тексты очищаются), заголовки безопасности на уровне Next/Trafik.
- Секреты только в k8s‑secrets; логи без приватных данных.

## 9. Производительность (минимальные цели)
- p95 API < 300ms при 20–50 rps, TTFB < 500ms для главной.
- Кэширование ответов справочников/цен (если применимо), соединения к Google API повторно используются.

## 10. CI/CD и инфраструктура
- GitHub Actions → сборка образа GHCR (теги: `minimal-stage`, `minimal-stage-{sha}`) → k8s (Traefik IngressRoute, cert‑manager, Let’s Encrypt).
- Namespace: `canton-otc-minimal-stage`, домен: `https://stage.minimal.build.infra.1otc.cc`.
- Пробы: liveness/readiness `/api/health`.

## 11. SLO/операционные требования (минимум)
- Доступность 99.5% в рабочие часы.
- MTTR < 30 мин.
- Время ручной обработки заявки оператором (операционный KPI) — до 12 часов.

## 12. Intercom — роль в минимальной версии
Intercom является основной точкой чата «клиент ↔ оператор». Клиент пишет в виджет Intercom на сайте; Спец‑бот (внутренний) ретранслирует сообщения в закрытую операторскую группу/канал с контекстом заказа (orderId, сумма, сеть, контакты). Ответы оператора возвращаются клиенту через Intercom.

### 12.1 События/данные, которые должны передаваться в Intercom
- `user_id`/`anonymous_id`, `orderId` (при наличии), контакт (email/телеграм/телефон), `utm`, язык/часовой пояс.
- Текущий `status` заказа и последние шаги (до 5 стадий).

### 12.2 Минимальные требования к интеграции
- Лёгкий виджет без деградации производительности; ленивое подключение после `load`.
- Грейсфул фоллбек: при ошибке загрузки — показать альтернативный контакт (телеграм/email).
- Событийная модель: отправка кастомных событий (создан заказ, ошибка валидации, клиент открыл чат).

### 12.3 Поток через спец‑бота
1) Клиент пишет в Intercom → Webhook/бот получает событие/текст.
2) Бот подтягивает контекст (orderId из cookie/localStorage/API) и публикует в операторскую группу: «Новая заявка/сообщение: …».
3) Ответ оператора из группы/бота идёт обратно клиенту в Intercom.
4) Запись в Google Sheets дополняется пометками переписки (опционально).

## 13. Критерии «Готово к бою» (минимум)
- Заявки создаются/логируются/видны в админке; Telegram‑уведомления приходят.
- Статусы обновляются и корректно отражаются клиенту.
- Intercom доступен и стабилен, сообщения доходят оператору через бота ≥ 99%.
- TLS валиден, домен доступен, редиректы корректны, /api/health «зелёный».

## 14. План тестирования (сжатый)
1) Инфраструктура: DNS→Ingress→TLS→/health.
2) Фронт: форма, статусы, ошибки в консоли = 0.
3) E2E заказ: создание→Sheets→Telegram→статусы.
4) Админка: вход, CRUD заказов/настроек.
5) Безопасность: лимиты/инъекции/заголовки.
6) Intercom: сообщение клиента→бот→оператор→ответ клиенту.
