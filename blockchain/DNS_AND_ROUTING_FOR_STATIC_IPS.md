# DNS записи и переадресация для статических IP адресов

## 📋 Краткий ответ

**Для валидации IP адресов Canton валидаторов:**

- ❌ **DNS записи НЕ нужны** - валидация идет по IP адресам напрямую
- ❌ **Переадресация портов НЕ нужна** - используются исходящие HTTPS соединения
- ✅ **Маршрутизация трафика (SNAT) нужна** - если IP адреса на одном сервере

---

## 🔍 Детальное объяснение

### 1. DNS записи

#### ❌ DNS записи НЕ требуются для валидации IP

**Почему:**
- Валидация IP адресов происходит через **прямые HTTPS запросы** к API Canton Network
- Используются IP адреса напрямую или доменные имена типа `scan.sv-1.testnet.global.canton.network.sync.global`
- Ваш IP адрес проверяется **Super Validators** (SV), которые видят ваш IP из исходящего трафика

**Что происходит при валидации:**
```bash
# Ваш сервер делает запрос:
curl https://scan.sv-1.testnet.global.canton.network.sync.global/api/scan/v0/scans

# Super Validator видит ваш IP адрес (65.108.15.20) в заголовках запроса
# И проверяет, есть ли этот IP в whitelist
```

**DNS записи могут понадобиться только если:**
- Вы хотите использовать доменные имена для доступа к валидатору (например, `testnet-validator.yourdomain.com`)
- Но для валидации IP это **не требуется**

---

### 2. Переадресация портов (Port Forwarding)

#### ❌ Переадресация портов НЕ нужна для валидации IP

**Почему:**
- Валидация IP - это **исходящие HTTPS соединения** от вашего сервера к Super Validators
- Вам не нужно принимать входящие соединения для проверки whitelisting
- Super Validators проверяют ваш IP по **исходящему трафику**

**Что нужно для валидации:**
```
Ваш сервер (65.108.15.20) 
    ↓ (исходящий HTTPS трафик)
Super Validator API (scan.sv-1.testnet.global.canton.network.sync.global)
    ↓ (проверяет IP 65.108.15.20 в whitelist)
Результат: ✅ или ❌
```

**Переадресация портов может понадобиться только если:**
- Вы хотите открыть доступ к валидатору извне (например, для health checks)
- Но для валидации IP это **не требуется**

---

### 3. Маршрутизация трафика (SNAT)

#### ✅ Маршрутизация трафика НУЖНА, если IP адреса на одном сервере

**Когда нужно:**
- Если IP адреса 65.108.15.20 (Testnet) и 65.108.15.19 (Mainnet) находятся на том же сервере, что и 65.108.15.30 (Devnet)
- Нужно убедиться, что исходящий трафик идет с **правильного IP адреса**

**Как это работает:**
```bash
# Без маршрутизации:
# Все запросы идут с основного IP (65.108.15.30)

# С маршрутизацией (iptables SNAT):
# Запросы к Testnet API → идут с IP 65.108.15.20
# Запросы к Mainnet API → идут с IP 65.108.15.19
```

**Настройка через iptables:**

```bash
# На сервере с несколькими IP адресами

# Для Testnet (65.108.15.20)
iptables -t nat -A OUTPUT \
  -p tcp \
  --dport 443 \
  -d scan.sv-1.testnet.global.canton.network.sync.global \
  -j SNAT --to-source 65.108.15.20

# Для Mainnet (65.108.15.19)
iptables -t nat -A OUTPUT \
  -p tcp \
  --dport 443 \
  -d scan.sv-1.global.canton.network.sync.global \
  -j SNAT --to-source 65.108.15.19

# Сохранить правила
iptables-save > /etc/iptables/rules.v4
```

**Автоматическая настройка:**
```bash
# Используйте готовый скрипт
bash blockchain/scripts/setup-multiple-ips-on-node.sh
```

## 🎯 Что нужно настроить на DNS провайдере

### ❌ Ничего не нужно настраивать на DNS провайдере

**Почему:**
1. **IP адреса уже назначены** - они статические и уже привязаны к вашему серверу
2. **DNS записи не влияют на валидацию** - валидация идет по IP адресам
3. **Переадресация не нужна** - используется исходящий трафик

**Единственное, что может понадобиться:**
- Если вы хотите использовать доменные имена для доступа к валидатору (опционально)
- Но это не влияет на валидацию IP адресов

---

## ✅ Что нужно настроить на сервере

### Если IP адреса на одном сервере:

1. **Добавить IP адреса на интерфейс:**
   ```bash
   # Через netplan или ip addr
   ip addr add 65.108.15.20/24 dev eth0
   ip addr add 65.108.15.19/24 dev eth0
   ```

2. **Настроить маршрутизацию (iptables SNAT):**
   ```bash
   # Используйте скрипт
   bash blockchain/scripts/setup-multiple-ips-on-node.sh
   ```

3. **Проверить, что IP видны:**
   ```bash
   ip addr show | grep "65.108.15"
   ```

### Если IP адреса на разных серверах:

1. **Подключить каждый сервер к кластеру:**
   ```bash
   # См. blockchain/CONNECT_STATIC_IPS_TO_CLUSTER.md
   ```

2. **Проверить подключение:**
   ```bash
   kubectl get nodes -o wide
   ```

## 🔍 Проверка правильности настройки

### Тест 1: Проверка IP адресов на сервере

```bash
# На сервере выполните:
ip addr show | grep "65.108.15"

# Должны увидеть все три IP:
# inet 65.108.15.30/24
# inet 65.108.15.20/24
# inet 65.108.15.19/24
```

### Тест 2: Проверка маршрутизации (если IP на одном сервере)

```bash
# Проверьте правила iptables
iptables -t nat -L OUTPUT -n -v | grep "65.108.15"

# Должны увидеть правила SNAT для Testnet и Mainnet
```

### Тест 3: Проверка исходящего IP

```bash
# Для Testnet
curl -sSL --interface 65.108.15.20 http://checkip.amazonaws.com
# Должен вернуть: 65.108.15.20

# Для Mainnet
curl -sSL --interface 65.108.15.19 http://checkip.amazonaws.com
# Должен вернуть: 65.108.15.19
```

### Тест 4: Запуск валидации

```bash
# Testnet
kubectl apply -f blockchain/k8s/ip-validation-testnet.yaml
kubectl logs -l job-name=ip-validation-checker-testnet -f

# Mainnet
kubectl apply -f blockchain/k8s/ip-validation-mainnet.yaml
kubectl logs -l job-name=ip-validation-checker-mainnet -f
```

## ❓ FAQ

### Q: Нужно ли настраивать A записи в DNS?

**A:** Нет, для валидации IP адресов DNS записи не нужны. Валидация идет по IP адресам напрямую.

### Q: Нужно ли настраивать переадресацию портов у провайдера?

**A:** Нет, для валидации IP переадресация портов не нужна. Используется исходящий HTTPS трафик.

### Q: Что делать, если IP адреса на одном сервере?

**A:** Используйте скрипт `setup-multiple-ips-on-node.sh` для автоматической настройки маршрутизации.

### Q: Что делать, если IP адреса на разных серверах?

**A:** Подключите каждый сервер к кластеру как отдельную ноду (см. `CONNECT_STATIC_IPS_TO_CLUSTER.md`).

### Q: Как проверить, что все настроено правильно?

**A:** Запустите проверку валидации IP через kubectl Jobs (см. раздел "Проверка правильности настройки").
